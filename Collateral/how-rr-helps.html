The Five Failure Points

Why manual reconciliation silently breaks.

The standard Stock Status report has five well-documented ways of producing a number that won't agree to the trial balance. Each one produces a mismatch that must be traced manually — and most recur every period if the root cause isn't found.

1
Timing
Server clocks vs. report runs
JD Edwards · Manual

A Stock Status report is run at a specific moment. Any transaction that posts after the report runs but before the period closes creates a discrepancy — the GL has the entry but the report does not.

In multi-timezone environments, transactions are recorded against the JDE server clock, not the user's local time. A user in a western time zone entering a transaction after the report runs can silently create a reconciling item.

RapidReconciler

Reconciliation is updated automatically with each nightly import cycle. There is no dependency on a perfectly timed manual report run.

Nightly import captures all posted transactions. Timing discrepancies are surfaced immediately, identified by document date and GL date.

2
Backdating
Period mismatches between physical and GL
JD Edwards · Manual

A transaction is posted to a GL period that differs from the period in which the inventory quantity actually changed. Stock Status reflects physical inventory; the GL reflects the backdated period.

Backdating can occur via manual date entry or via processing options in Manufacturing Accounting and Sales Update — sometimes intentionally, sometimes not. Either way, Stock Status and trial balance won't agree.

RapidReconciler

Backdated transactions are captured with both their transaction date and GL date visible, making the period mismatch immediately identifiable.

The Transactions page surfaces period mismatches as a labeled variance type. The Sub Type field identifies "Periods" as the source — no manual tracing required.

3
Report Definition
Cost methods, data selection, inclusion logic
JD Edwards · Manual

The Stock Status report depends on correct cost method logic, data selection criteria, and inclusion rules. These drift over time as items, cost methods, and business units change.

  • Hardcoding cost method "07" instead of using COCSIN = 'I'
  • Including non-stock items that carry an inadvertent cost
  • Excluding stock items through incorrect data selection
RapidReconciler

RapidReconciler reads directly from JD Edwards data tables — F4111, F0902, F0911 — eliminating the risk of incorrect cost method logic or flawed data selection in a custom report.

No report definition to configure. The reconciliation is driven by the same source tables JD Edwards uses. If an item has a cardex record, it is included.

4
DMAAI Misconfiguration
Silent transaction-to-GL mismaps
JD Edwards · Manual

DMAAI tables define how inventory transactions map to GL accounts. An incorrect account number causes every transaction of that type to post to the wrong account — silently, with no error message.

Unlike timing or backdating issues, DMAAI misconfiguration is entirely invisible until reconciliation. Tracing it requires hours of manual review across DMAAI tables, document types, and GL class codes.

RapidReconciler

Integrity Reports compare DMAAI entries against the model table and flag mismatches by object account, business unit, and subsidiary. The Transaction Detail report identifies the specific AAI responsible.

Configuration issues are surfaced proactively by six Integrity Reports that run on every import cycle. A misconfigured AAI is identified before it compounds into a large period-end variance.

5
GL Class Code Changes
Stranded inventory value in old accounts
JD Edwards · Manual

Changing a GL class code on an item with quantity on hand does not automatically generate a journal entry to reclassify inventory value. Subsequent transactions post to the new account while the original value remains stranded in the old account.

JDE allows GL class code changes at any level without checking whether inventory is on hand. The result: a permanent reconciling item that grows with every subsequent transaction.

RapidReconciler

GL class code mismatches are surfaced immediately. Variances appear in the reconciliation on the next import cycle. Integrity Report 5 identifies mismatches between Item Branch and Item Location records.

The impact is visible before it compounds. Integrity Report 3 flags GL class codes missing from the model. The As-Of page surfaces items carrying balances in multiple accounts.

Inventory Reconciliation

F4111 perpetual ledger ↔ F0902 / F0911 GL

The Challenge

The perpetual inventory balance and the GL balance should always agree — but timing differences, DMAAI mismatches, cardex integrity issues, unposted batches, and GL class code changes each create variances that the standard Stock Status report cannot identify, isolate, or explain.

FeatureWhat it does
Valuation Section ✓Automatically compares summarized F4111 perpetual balance to F0902 GL balance for every period — no manual report run required. If the out-of-balance is zero, inventory is reconciled.
Variance Calculation ✓Breaks the total out-of-balance into six labeled sources — Carry Forward, GL Batches, End of Day, Transactions, Cardex, and Manual Journal Entries — each with the correct resolution path.
Cardex Integrity Pop-Up ✓Compares summarized F4111 to F41021 for every item, excluding memo transactions and applying UOM conversions. Shows which items have variances and whether the issue is quantity (requires IT) or dollars (requires a dollars-only IA adjustment).
Transaction Detail Drill-Down ✓For any unreconciled transaction, shows the F4111 cardex, F0911 GL entries, how RapidReconciler matched them, and the complete DMAAI entries for every GL class code in the transaction. Identifies the exact AAI causing an account mismatch.
As-Of Page ✓Continuously maintained period-end inventory position by item, branch, location, and lot — with QtyVar and AmtVar columns that surface cardex integrity issues at the item level. Replaces the timing-dependent Stock Status report entirely.
Re-Roll Options ✓Three tools to synchronize RapidReconciler with JD Edwards after a correction — Re-Roll Item, Zero Beginning Balance, and Remove CX Var — with period recalculation and full audit transparency.
Audit Report ✓Complete period-end reconciliation document — account summaries, unposted batches, open orders, manual entries, variances, and perpetual detail — exportable in Excel or PDF.

In Transit Reconciliation

ST/OT transfer activity ↔ In Transit clearing account

The Challenge

When goods move between branch plants via ST/OT transfer orders, their value sits in an In Transit clearing account until receipt. Standard cost changes, cost differences between branches, and quantity mismatches all create balances that don't clear. Once both orders reach status 999, no further JDE processing is possible.

FeatureWhat it does
Orders Page ✓Automatically calculates the in-transit position (Shipments − Receipts) for every order pair. Orders that reconcile exactly are removed automatically — no manual filtering required.
Exclusion Process ✓When both ST and OT orders are closed (status 999) but a residual balance remains, the Exclusion feature isolates those amounts and displays the exact journal entry value needed to clear the In Transit GL account.
ExclVarQty / ExclVarAmt ✓Monitors previously excluded orders for new activity. If a receipt is processed against an excluded open PO, these columns surface the discrepancy immediately — preventing new transactions from being silently omitted.
Variance Calculation ✓Breaks the GL out-of-balance into Carry Forward, GL Batches, End of Day, Transactions, Exclusions, and Manual Journal Entries — each with the appropriate resolution path identified.
As-Of Page ✓Reconstructs the in-transit position as of any historical period end, calculated backwards from the current position. Full transaction-level detail with drill-down and export for period-end documentation and audit support.
Integrity Report 4 ✓Lists orders where the ST sales cost or price does not match the OT purchase order cost — the most common cause of In Transit balances that do not clear after receipt.

PO Receipts (RNV) Reconciliation

F43121 receipt table ↔ Received-Not-Vouchered GL account

The Challenge

The F43121 table is not a true ledger. Receipt and voucher reversals overwrite existing records rather than creating new ones, making As-Of reporting impossible. Reconciliation must always be a current balance-to-balance comparison — meaning manually matching Match Type 1 and 2 records across hundreds of PO lines, every period.

FeatureWhat it does
Orders Page ✓Imports F43121 and F0911 automatically and displays only purchase orders with open amounts or variances between the receipts table and the GL. Fully reconciled orders are removed automatically.
Calculation Section ✓Separates the out-of-balance into Open Receipts, Unreconciled, and Batches components. Provides a Suggested Entry amount excluding outstanding variances.
PO Receipts Aging ✓Visual breakdown of open receipts by period — immediately identifying whether the RNV balance reflects recent legitimate activity or aged receipts that have not been vouchered.
Suspension Feature ✓Removes orders cleared by manual voucher entry, accrual-only landed costs (PRLAND = 3), or data cutoff artifacts from the variance calculation — isolating genuine unresolved items from known exceptions.
Line Analysis Page ✓Side-by-side F43121 vs. GL comparison at the document level for any PO line. Supports document-level exclusion with audit notes and a Recalc function that updates variance calculations within minutes.
Document Button ✓Displays all transactions for a PO line with a Variance column identifying precisely which documents have discrepancies between F43121 and F0911.
Proactive Monitoring

Six Integrity Reports. Run every cycle.

A single misconfigured AAI silently routes every transaction of that type to the wrong GL account. The Integrity Reports surface configuration issues before they compound into large period-end variances.

At Scale

Why the manual approach breaks down.

Each issue is manageable in a small, simple JD Edwards environment. The manual approach breaks down as complexity increases — and complexity always increases.

↗

Volume

As transaction volume grows, the number of potential reconciling items grows with it. Investigation that takes an hour for 10 items takes a day for 100.

⊞

Multiple Companies

Each company requires its own Stock Status and trial balance comparison. DMAAI configuration must be maintained independently per company. Errors in one are invisible to reconcilers working on another.

$

Multiple Cost Methods

Environments mixing standard cost, weighted average, and actual cost require careful report definition management. A single incorrect cost method affects every item using it.

↻

Frequent Cost Changes

Standard cost updates can create WIP revaluation variances. Without a tool to surface these, they accumulate as unexplained balances until the next reconciliation cycle.

⤺

Staff Turnover

The manual process depends on institutional knowledge of report configuration, DMAAI setup, and backdating practices. When the person who knows leaves, that knowledge leaves with them.

✓

A consistent, automated reconciliation methodology that does not depend on report configuration, server timing, or organizational knowledge of DMAAI setup. It just runs.

RapidReconciler Design Philosophy
Important to Know

What RapidReconciler is — and isn't.

RapidReconciler is a reconciliation and diagnostic tool. Understanding its boundaries helps set accurate expectations and ensures the right workflows are in place.

🔒

Read-only access to JD Edwards

RapidReconciler reads JD Edwards data in read-only mode. It never modifies JD Edwards tables. All corrections — adjustments, journal entries, AAI changes — are made directly in JDE.

🌙

Nightly data refresh

Data is imported from JD Edwards on a nightly cycle. Transactions entered after the most recent import will not appear in RapidReconciler until the following night's refresh.

📐

Identifies, not corrects

RapidReconciler identifies what is out of balance, where the discrepancy originates, and what the correct corrective action is. The resolution itself always happens in JD Edwards.

📅

History from initiation date

RapidReconciler calculates its cardex position from the point the program was first initiated, or from the last data reset. Transaction history predating that point is not visible without a Re-Roll.
