How to Use This Document

Each scenario block contains five elements:

  • Scenario title and role badge — identifies the issue and who owns the resolution.
  • Symptom — what the customer or you will observe.
  • Likely causes — the most common reasons this symptom occurs.
  • Troubleshooting steps — numbered procedure to remediate, with a Copy button to copy the full text to your clipboard for pasting into tickets or chats.
  • Related — deep-links into the source procedure documents for full context.
Principle

If the symptom doesn’t match any scenario in this document, the agent log files (see Diagnostic Tools) are almost always the next place to look. The full agent log usually contains more context than any single error message a customer will quote.

Quick Triage

When a customer reports “RapidReconciler isn’t working,” the fastest triage path is usually a three-step pass through the VALC Clients grid. The result tells you which category of scenario in this document to consult first.

The Three-Step Triage

1
Start here
Check the customer’s row on the VALC Clients grid

Look at the four indicators from left to right: Status badge, Services version, Agent version, System Status. The leftmost Status column tells you if the agent is currently connected to VALC. The System Status column tells you if the most recent JDE-to-RR data sync succeeded.

These two indicators fail in different ways and require different fixes — mixing them up sends you down the wrong path.

2
Then check
Open Manage Client → Databases tab

Confirm every registered database row shows Online (green). Anything else (Unknown / Stopped) is the #1 reason “the customer can’t see their data.”

3
Decide path
Decide which category to consult

Both Status and Databases healthy → the issue is on the JDE-side data flow. Check Database & SSIS first (especially the SQL Agent job and SSIS package).

Status unhealthy → agent connectivity. Check Network & Connectivity and Certificate & DNS.

System Status unhealthy → data sync. Check Database & SSIS.

Database tab unhealthy → Database Status: Unknown or Stopped.

Customer reports “I can log in but I don’t see any data” — and everything in VALC looks healthy from the GSI side → the issue is almost certainly customer-side. Walk them through the customer-side checks (VPN, hard reload, browser local network access, firewall, browser cache) at the top of Network & Connectivity.

Install & Agent

Issues that arise during agent installation or immediately after — the agent download prompt, the install flow itself, and the post-install verification.

Agent install hangs at "Validating Data"

RR Network Tech
Symptom: During agent install, the browser displays "Validating Data" indefinitely instead of progressing to the SQL Server connection prompt or "Deploying".
Likely causes
  • Application server cannot reach the database server on the configured SQL port
  • SQL credentials provided in the connection prompt are incorrect
  • RapidReconciler_Prod database is not yet created on SQL Server, or schema is incomplete
  • Firewall on the database server is blocking inbound traffic on port 1433 (or the customer's configured port)
  • SQL Server Browser service is not running on the database server
Troubleshooting steps
  1. From the application server, attempt a manual connection to the database server on the configured port to verify reachability.
  2. Confirm the database, the rruser login, and SQL Authentication mode are all in place per the pre-install database setup.
  3. If the prompt is reached but still fails, double-check the values being entered match the customer's SQL configuration exactly.
  4. Check Windows Event Viewer on the application server for RapidReconciler-related errors.
  5. Check today's agent .out and .err log files (see Diagnostic Tools).
  6. If the install needs to be restarted, ensure Setup Step is set back to "Created" in VALC before retrying.
In coordination with Customer I/T (server access)
Related: Installing a Client in VALC → Part 2: Installing the Agent

Agent download prompt does not appear after login

RR Network Tech
Symptom: Customer logs in to rapidreconciler.getgsi.com with GSIADMIN credentials but no agent download prompt is shown.
Likely causes
  • Setup Step on the Client Details tab is not set to "Created"
  • Login was performed with the wrong credentials
  • Browser cache is showing a stale view
Troubleshooting steps
  1. In VALC, open Manage Client → Client Details tab for the customer.
  2. Verify Setup Step is set to "Created". If not, change it to "Created" and save.
  3. Have the customer hard-refresh the browser (Ctrl+F5 / Cmd+Shift+R) and log in again with the GSIADMIN credentials.
  4. If the prompt still does not appear, check that no other browser session has the customer logged in elsewhere.
Related: Installing a Client in VALC → Setup Step Not Advancing

Clients grid row not updating to "Connected" after install

RR Network Tech
Symptom: Agent install reported "Installation Complete", but on the VALC Clients grid the row's Status badge stays at the previous value instead of switching to Connected.
Likely causes
  • DNS for the customer's .getgsi.com subdomain has not yet propagated after Cloudflare update
  • RR Agent and RR Services are installed but one or both services are not running
  • VALC has not refreshed — the underlying agent has connected but the grid row hasn't updated yet
  • Agent External IP on the Client Details tab does not match the customer's actual external IP
Troubleshooting steps
  1. Refresh the Clients grid in your browser (the page does not auto-refresh).
  2. Wait a few minutes if the install just completed.
  3. From the customer's application server, try resolving the customer's .getgsi.com URL — if DNS still returns the old IP, the Cloudflare update hasn't propagated yet.
  4. On the application server, confirm both "RapidReconciler Agent" and "RapidReconciler Services" are running in the Windows Services console.
  5. Verify the Agent External IP Address on the Client Details tab matches the customer's actual external IP.
  6. If the row still hasn't updated, check today's agent .out log file for connection errors back to VALC.
Related: Installing a Client in VALC → Cloudflare Update but Data Not Appearing

Setup Step not advancing automatically after install

RR Network Tech
Symptom: After completing an install or migration, the Setup Step field on the Client Details tab remains at "Created" (or an earlier value) and does not advance.
Likely causes
  • The install did not complete fully — agent, database connectivity, or services may have an unresolved issue
  • The state genuinely needs to be moved manually for this install path
Troubleshooting steps
  1. Confirm the install actually completed end-to-end: agent installed, database showing Online, connectivity row showing Connected.
  2. If all of the above are good, manually advance the Setup Step on the Client Details tab to the appropriate next value.
  3. If you're not sure what value to set, check another active client for reference.
  4. Save the change and refresh the Clients grid to confirm the row updates.
Related: Installing a Client in VALC → Setup Step Not Advancing

Database & SSIS

Database registration, SSIS package configuration, SQL Agent job, and the JDE source connection. Most of these scenarios are owned by the RR DBA.

Database status shows "Unknown" or "Stopped"

RR DBA
Symptom: On the Databases tab, a row shows Unknown (orange) or Stopped (orange) instead of Online (green).
How to tell apart: Stopped usually means the database has never connected since being registered (Revision Number is blank). Unknown usually means the agent had previously reached the database but has since lost contact (Revision Number is populated).
Likely causes
  • Connection details on the Edit Database popup are wrong — IP, port, username, password, or name don't match the actual SQL Server
  • RR Agent service on the customer's application server is stopped or has lost network access
  • Firewall on the database server is blocking the agent's connection on the configured port
  • The physical database doesn't exist on SQL Server (registration in VALC succeeded but the underlying database wasn't created or has been removed)
  • SQL Server Authentication is disabled or the rruser login was not created
Troubleshooting steps
  1. Open the Edit Database popup for the failing row in VALC and verify every field matches the actual database configuration.
  2. Confirm the RR Agent service is running on the customer's application server. If not, start it.
  3. Check today's agent .out log file on the application server for connection errors or stack traces.
  4. From the application server, attempt a manual connection to the database server on the configured port (e.g., telnet 10.x.x.x 1433) to confirm reachability.
  5. Confirm the customer's DBA has created the RapidReconciler_Prod database and the rruser SQL Authentication login.
  6. If everything looks correct and status remains failed, escalate to the GSI database administrator.
In coordination with RR Network Tech (VALC) and Customer I/T (server access)
Related: Installing a Client in VALC → Database Status: Unknown or Stopped · Installing the Production Database

Companies tab is empty after install

RR DBA
Symptom: Database shows Online on the Databases tab, but selecting it from the dropdown on the Companies tab shows no company numbers to license.
Likely causes
  • The first sync from JDE has not yet completed — Online status confirms VALC can reach the database, but company numbers populate from JDE through the SSIS package's first run
  • The SQL Agent job is not running or is disabled on the database server
  • The SSIS package's source connection (to JDE) is misconfigured, even though the destination (to RapidReconciler_Prod) is working
  • The customer's JDE environment has no companies in the source tables that the SSIS package reads
Troubleshooting steps
  1. On the Companies tab, check the "Last Company Sync" timestamp at the top-right. If it shows recent activity, give the sync a few more minutes and refresh the page.
  2. On the database server, verify the RapidReconciler SQL Agent job is enabled and has run successfully at least once.
  3. Check the SQL Agent job history for errors on the most recent run.
  4. If the job is failing or has never run, escalate to the GSI database administrator to investigate the SSIS package and JDE connection.
Related: Installing a Client in VALC → Part 3 Step 3: License Company Numbers

SSIS package deployed but extracts no data

RR DBA
Symptom: SSIS package was deployed successfully and the SQL Agent job runs without errors, but no data flows into RapidReconciler_Prod.
Likely causes
  • The dbowner field is missing the trailing period (e.g. "proddta" instead of "proddta.") — the single most common cause of this ticket type
  • SSIS package source connection points to the wrong JDE environment
  • The customer's JDE source tables genuinely have no data in the queried range
Troubleshooting steps
  1. In the SSIS package configuration, open the dbowner parameter value and verify the trailing period is present (e.g. "proddta." not "proddta").
  2. Verify the SSIS package source connection string points to the correct JDE database for this customer.
  3. Run the SSIS package manually with verbose logging and inspect which step finds zero rows.
  4. If the source connection is correct and the package finds no rows, confirm with the customer that the queried JDE tables are populated.
Related: Installing the Production Database → Tour 3: Configure & Deploy SSIS Package

SSIS package deployment fails — OLE DB driver error

RR DBA
Symptom: During SSIS package deployment, an error indicates the OLE DB provider for JDE is not installed or cannot be found.
Likely causes
  • Driver mismatch — only 64-bit installed when both 32-bit and 64-bit are needed
  • OLE DB provider was not installed on the database server
  • Wrong version of the OLE DB provider for the JDE database type
Troubleshooting steps
  1. On the database server, verify both 32-bit and 64-bit versions of the required OLE DB provider are installed.
  2. Re-install the OLE DB provider with both architectures if missing.
  3. After installation, re-run the SSIS package deployment.
  4. Verify driver installation before the install meeting next time — this is a known cause of install delays.
Related: Installing the Production Database → Pre-install Requirements

SSIS catalog deployment fails — permission denied

RR DBA
Symptom: When deploying the SSIS package to the SSISDB catalog, deployment fails with a permission/authentication error.
Likely causes
  • The deploying account does not have sysadmin or ssis_admin privileges
  • Windows Authentication was not used for deployment (required)
  • SSISDB catalog has not been created on the target SQL Server instance
Troubleshooting steps
  1. Confirm you are deploying using Windows Authentication, not SQL Authentication.
  2. Verify the deploying account has sysadmin or ssis_admin privileges on the target SQL Server instance.
  3. Confirm the SSISDB catalog exists — if not, create it first.
  4. If the SQL Agent job steps will run under a proxy account, ensure the proxy has access to any UNC paths used by the package.
Related: Installing the Production Database → Tour 3: Configure & Deploy SSIS Package

Network & Connectivity

Network-side issues. The first five scenarios are customer-side first-line checks for “I can log in but I don’t see any data” — the questions to walk a customer through when they call support. The remaining scenarios cover GSI-side issues: Cloudflare changes and agent service problems on the customer’s application server.

Customer logs in but sees no data — working remotely without VPN

RR Network Tech
Symptom: Customer reports they can log in to RapidReconciler successfully, but no data is displayed. They are working from home, a coffee shop, or an off-site location.
Likely causes
  • Customer is not connected to their corporate network
  • Customer is not connected via VPN or another approved remote-access method
  • VPN connection dropped during the session
Troubleshooting steps
  1. Ask the customer if they are working remotely (home / travel / off-site).
  2. If remote, ask them to confirm they are connected via VPN or their organization's approved remote-access method.
  3. If they are not connected, ask them to connect to VPN and try again.
  4. If they believe they are connected, ask them to disconnect and reconnect the VPN, then hard-refresh the browser.
  5. If the issue persists after confirmed VPN connection, proceed to the next customer-side check (hard reload).
First-line response. Confirm with customer before escalating.

Customer logs in but sees no data — first-pass hard reload

RR Network Tech
Symptom: Customer reports they can log in but no data is displayed. They are confirmed on the corporate network or VPN.
Likely causes
  • Browser is showing a stale cached version of the page
  • A transient connection issue caused the initial page load to fail partially
  • Browser session needs to re-authenticate against the agent
Troubleshooting steps
  1. Ask the customer to perform a hard reload of the browser using Shift + F5 (Windows) or Cmd + Shift + R (Mac).
  2. If that doesn't resolve it, ask them to fully close all browser windows and reopen, then log in again.
  3. If still no data appears, proceed to the browser local network access check.
First-line response. The cheapest check to ask the customer to perform.

Browser blocking local network access (Edge / Chrome permission)

RR Network Tech
Symptom: Customer logs in successfully but no data displays. VPN is confirmed connected and a hard reload didn't help. Customer's browser was recently updated, or this is a new workstation.
Likely causes
  • Recent browser updates introduced a permission controlling whether sites can connect to devices on the local network
  • rapidreconciler.getgsi.com is not in the allowed list for local network access
  • A workstation policy is preventing the user from changing the setting
Troubleshooting steps
  1. Walk the customer through Edge: Settings → Privacy, search, and services → Site permissions → All permissions → Local network access.
  2. In the "Customized behaviors" section, check the "Allowed to connect to any device on your local network" list.
  3. Verify rapidreconciler.getgsi.com is present in that allowed list.
  4. If missing, click "Add site" next to that list and add rapidreconciler.getgsi.com.
  5. Chrome has an equivalent setting under its site-permissions area — wording differs slightly but the concept is the same.
  6. After adding the site, hard-reload the page and verify data appears.
  7. If the customer cannot find or change the setting on their machine, escalate to their I/T department.
Coordinate with Customer I/T if customer can't change the setting themselves.

Corporate firewall or IP filter blocking RapidReconciler

Customer I/T
Symptom: Customer logs in but no data displays. The issue began suddenly with no change on the customer's end. Multiple users at the same customer report the same issue.
Likely causes
  • Customer's I/T team recently updated the corporate firewall
  • New IP filtering software was rolled out across the customer's network
  • A security policy change blocked outbound traffic to rapidreconciler.getgsi.com
Troubleshooting steps
  1. Ask the customer if their I/T team recently made changes to the firewall, deployed new security software, or rolled out IP filtering.
  2. If yes, ask the customer to contact their I/T team and confirm that rapidreconciler.getgsi.com is permitted through any firewall rules or filter lists applying to their workstation.
  3. When multiple users at the same customer are affected simultaneously, this scenario is the most likely cause — escalate directly to Customer I/T.
  4. After confirmation from Customer I/T that the domain is allowed, have the customer hard-refresh the browser and verify data appears.
Diagnosed by RR Network Tech, remediated by the customer's I/T team.

Customer logs in but sees no data — last-resort browser data clear

Customer I/T
Symptom: Customer logs in but no data displays. VPN is confirmed, hard reload didn't help, local network access is correctly configured, and Customer I/T has confirmed no firewall changes.
Likely causes
  • Browser cache contains stale or corrupted authentication tokens
  • Cookies for the rapidreconciler.getgsi.com domain are in an invalid state
  • Service worker or local storage holds outdated state from a prior session
Troubleshooting steps
  1. Refer the customer to their I/T department for browser-specific instructions on clearing browser data.
  2. The general approach is to clear cookies and cached files for rapidreconciler.getgsi.com (not all sites), then close the browser fully and reopen.
  3. After clearing, have the customer log in again with a fresh session.
  4. If the issue still persists after all customer-side checks, the issue is on the GSI side — proceed to the agent-side scenarios in this document and check the agent connectivity from the customer's server.
  5. If exhausted all checks and no resolution, email rrsupport@getgsi.com with the customer details and the steps already attempted.
Procedure varies by browser; refer the customer to their I/T department for browser-specific instructions.

Cloudflare update made but data not appearing in UI

RR Network Tech
Symptom: Cloudflare has been updated by GSI I/T and confirmed propagated, but data still doesn't appear in the customer's RapidReconciler UI.
Likely causes
  • SQL Agent job schedule is not yet enabled (intentionally deferred until after Cloudflare confirmation in migrations)
  • SQL Agent job is enabled but has not yet had a chance to run on its schedule
  • SSIS package connection strings still point to the old server (after a migration)
  • Browser cache or session is showing stale data
Troubleshooting steps
  1. On the database server, confirm the SQL Agent job schedule is enabled.
  2. Trigger a one-time manual run of the SQL Agent job to validate it works.
  3. Check the job history for errors on the most recent run.
  4. Have the customer log out and log back in, or hard-refresh their browser.
  5. If the issue is post-migration, double-check SSIS package connection strings on the new server.
In coordination with RR DBA (SQL Agent job) and GSI I/T (Cloudflare)
Related: Installing a Client in VALC → Cloudflare Update Made but Data Not Appearing · Server Migration

Agent services are not running on application server

RR Network Tech
Symptom: In Windows Services console, "RapidReconciler Agent" or "RapidReconciler Services" is shown as Stopped, or fails to start.
Likely causes
  • Service was stopped manually or by Windows Update reboot
  • Service account credentials are invalid or expired
  • Configuration file was modified or corrupted
  • A required dependency service is also stopped
Troubleshooting steps
  1. On the application server, open Services console (services.msc).
  2. Right-click "RapidReconciler Agent" and click Start. Repeat for "RapidReconciler Services".
  3. If the service fails to start, check today's agent .wrapper log file for the startup error.
  4. Verify the service's log-on credentials are still valid.
  5. Check Windows Event Viewer (Application log) for related errors at the time of failure.
  6. If the wrapper log indicates a config issue, escalate to a senior tech to investigate the agent installation.
In coordination with Customer I/T (server access)
Related: Diagnostic Tools — Agent Log Files

Certificate & DNS

SSL certificate warnings, DNS resolution failures, and IP/DNS mismatches. Most of these route through GSI I/T (Daren) for the actual change.

Browser shows "Not Secure" or certificate warning

RR Network Tech
Symptom: When a customer navigates to their RapidReconciler URL, the browser shows a security warning or "Not Secure" indicator.
Likely causes
  • Wildcard certificate has expired
  • Renewed certificate has not yet been deployed to the customer's agent
  • UI Developer has not yet updated the backend configuration with the renewed certificate
Troubleshooting steps
  1. Check the certificate expiry date in the browser by clicking the lock/warning icon in the address bar.
  2. If expired or close to expiry, verify with GSI that the wildcard *.getgsi.com cert renewal is complete.
  3. If renewal is complete but deployment hasn't happened, redeploy the agent for this customer via VALC Manage Deploys.
  4. Verify in the browser that the new certificate is active after redeployment.
  5. For systemic issues across many customers, confirm the UI Developer has updated the backend configuration.
Escalate to GSI Staff for VALC Manage Deploys; UI Developer for backend updates
Related: SSL Certificate & DNS Management → SSL Renewal

Customer's domain URL does not resolve

GSI I/T (Daren)
Symptom: Customer cannot reach their RapidReconciler URL; the domain returns "DNS_PROBE_FINISHED_NXDOMAIN" or similar resolution failure.
Likely causes
  • DNS A record is missing or incorrect
  • A record was never created for this customer
  • A record points to a wrong IP after a server change
Troubleshooting steps
  1. In VALC, open Manage Client → Client Details and confirm the Domain URL value.
  2. Look up the A record for this domain (e.g. via dig or nslookup) and verify it returns the customer's correct internal IP.
  3. If the record is missing or wrong, contact GSI I/T (Daren) to verify and correct the DNS entry.
  4. After correction, allow time for DNS propagation before testing again.
Initiated by RR Network Tech
Related: SSL Certificate & DNS Management → DNS Configuration

UI loads but agent is not connecting

RR Network Tech
Symptom: Customer can reach the RapidReconciler UI at their domain URL, but the VALC Clients grid shows the agent as Disconnected, or no data appears in the UI.
Likely causes
  • Customer's internal IP address has changed but the change is not reflected in VALC
  • IP change is in VALC but DNS still points to the old IP
  • Agent service is stopped on the application server
Troubleshooting steps
  1. In VALC, verify the Agent Internal IP Address on the Client Details tab matches the customer's actual current IP.
  2. If different, update the IP in VALC.
  3. Submit a DNS update request to GSI I/T (Daren) with the corrected internal IP.
  4. On the application server, verify the RR Agent service is running.
  5. After DNS propagation, confirm the Clients grid row updates to Connected.
In coordination with GSI I/T (Daren) for DNS updates
Related: SSL Certificate & DNS Management → Common Issues

DNS propagation taking longer than expected

GSI I/T (Daren)
Symptom: A DNS update was made by GSI I/T (Daren) several hours ago but the change is still not visible to the customer.
Likely causes
  • ISP TTL settings are causing slow propagation
  • Customer's local DNS cache is holding the old value
  • Intermediate caching DNS resolver in the customer's network
Troubleshooting steps
  1. Allow additional time — propagation can take up to 24 hours depending on TTL settings.
  2. Have the customer flush their local DNS cache (ipconfig /flushdns on Windows).
  3. If the issue persists beyond 24 hours, contact GSI I/T (Daren) to verify the DNS change is visible at the authoritative nameserver.
  4. Test from a different network to rule out customer-side DNS caching.
Diagnosed by RR Network Tech
Related: SSL Certificate & DNS Management → DNS Configuration

Server Migration

Migration-specific gotchas. Standard install scenarios in the other categories also apply during a migration; this category captures the issues unique to the cutover process.

Agent download prompt does not appear during migration

RR Network Tech
Symptom: On the new server during migration, after logging into rapidreconciler.getgsi.com with GSIADMIN credentials, no agent download prompt is shown.
Likely causes
  • Setup Step is not set to "Created" in VALC for this customer
  • Existing agent registration on the old server is interfering with the prompt
Troubleshooting steps
  1. In VALC, open Manage Client → Client Details for the migrating customer.
  2. Change Setup Step to "Created" and save.
  3. Hard-refresh the browser on the new server (Ctrl+F5) and log in again with GSIADMIN credentials.
  4. The agent download prompt should now appear.
Related: Server Migration

Migration complete but data is not flowing on the new server

RR DBA
Symptom: After migration cutover, the customer can reach the RapidReconciler UI on the new server, but no new data appears.
Likely causes
  • SQL Agent job schedule was not re-enabled on the new server (intentionally disabled during migration setup)
  • SSIS package connection strings still point to the old database server
  • Cloudflare A record still points to the old server's IP
  • DNS propagation has not yet completed
Troubleshooting steps
  1. On the new database server, verify the SQL Agent job schedule is enabled. If not, enable it.
  2. Trigger a one-time manual run of the SQL Agent job to confirm it works on the new server.
  3. Open the SSIS package on the new server and verify all connection strings point to the new server, not the old one.
  4. Verify with GSI I/T (Daren) that the Cloudflare A record has been updated to the new server's IP and propagation is complete.
  5. Have the customer hard-refresh their browser to clear any cached UI state.
In coordination with RR Network Tech and GSI I/T (Daren)
Related: Server Migration

Diagnostic Tools

The primary diagnostic tool for any RapidReconciler issue is the agent log files on the customer’s application server. Whenever a scenario in this document says “check the agent logs,” this is what it’s referring to.

Agent Log Files

The RapidReconciler Agent writes detailed log files on the customer’s application server. These are the source of truth for diagnosing connectivity issues.

Log file location

C:\Program Files\Rapid Reconciler\logs\

What each log tells you

FileWhat’s in itWhen to use it
rr-valc-agent_YYYYMMDD.out Daily output log of the agent’s normal activity, including connection attempts and successes. First file to open. Most connectivity diagnostics live here. Open today’s file (matching the current date) and scroll to the bottom for recent entries.
rr-valc-agent_YYYYMMDD.err Daily error log. Stack traces, exceptions, fatal errors. Inspect alongside the .out file when the agent is reporting issues.
rr-valc-agent.wrapper Service wrapper log — tells you whether the agent service itself is starting properly. Use when the RR Agent service won’t start at all (Stopped in Services console).
Working remotely

If you’re diagnosing remotely, ask the customer’s I/T contact to send you the latest .out file rather than transcribing errors over a call. The full log usually contains more context than any single error message they’ll quote.

Other Diagnostic Tools

ToolWhat to check
Windows Services console (services.msc) Are RapidReconciler Agent and RapidReconciler Services running on the application server?
Windows Event Viewer (Application log) RapidReconciler-related errors, especially when the agent service won’t start. Cross-reference with the .wrapper log.
SQL Agent job history (database server) Did the most recent run of the RapidReconciler SQL Agent job succeed? Errors here mean the SSIS package is failing.
telnet [db-server-ip] [port] Tests TCP connectivity from the application server to the database server on the SQL port. If telnet fails, the agent can’t reach the database either.
nslookup [customer].getgsi.com Verifies what IP the customer’s domain currently resolves to. Compare against the IP in VALC.
ipconfig /flushdns (customer-side) Clears the local DNS cache. Useful when DNS has been updated but the customer still hits the old IP.
