Overview

Orientation for the pre-install process — what you'll do, what GSI handles, and how the steps flow together.

What You'll Do

This document walks you through preparing your environment for RapidReconciler installation. The customer's IT team is responsible for steps 1–5; GSI handles the install meeting itself once you submit your information.

Pre-Install Process

  1. Choose a deployment topology. Three options are supported; pick the one that best fits your environment.
  2. Identify your JD Edwards source platform. Oracle, AS/400 (I-Series), or Microsoft SQL Server. Determines which connectivity drivers SSIS will need.
  3. Provision and configure the required servers. Hardware, OS, SQL Server, Integration Services, and the supporting tools.
  4. Apply Windows security and network rules. Service accounts, antivirus exclusions, firewall rules, and outbound access to GSI hosts.
  5. Work through the pre-install checklist as you complete each step. Mark items as done.
  6. Gather JD Edwards connection details and submit them to GSI using the email button at the end of this document, along with your topology and source platform selections and checklist results. GSI will then schedule the install meeting.

Oracle — Driver Guidance

SSIS connects to Oracle through the Oracle Provider for OLE DB, which is delivered with the Oracle client software. The provider is installed by the Oracle client installer; it is not part of Windows or SQL Server.

What to Install

  • Oracle ClientInstall the Oracle Database Client (or Oracle Data Access Components — ODAC) using the Administrator install option, which includes the OLE DB provider. Match the client major version to your Oracle server version where possible.
  • ArchitectureInstall both 32-bit and 64-bit versions on the SSIS server. The 32-bit client is required by Visual Studio at design time; the 64-bit client is required when packages execute under the 64-bit SQL Agent runtime.
  • Oracle HomeInstall both architectures into the same Oracle home (or symmetric homes with the same name on each architecture). Mixing homes is a frequent source of "provider not registered" errors.
  • Provider NameIn SSIS connection managers the provider appears as Oracle Provider for OLE DB (registered as OraOLEDB.Oracle). Do not use the legacy Microsoft OLE DB Provider for Oracle — it is deprecated and based on Oracle Call Interface version 7.

Configuration After Install

  1. Add an entry for your JDE Oracle instance to tnsnames.ora. The file lives under the network/admin folder of each Oracle home — typically %ORACLE_HOME%\network\admin\tnsnames.ora. Both the 32-bit and 64-bit homes need the entry (or a shared TNS_ADMIN environment variable pointing to a single file).
  2. Set the TNS_ADMIN environment variable if you want both homes to read from one file.
  3. Restart the server (or at minimum the SQL Server and SSIS services) after install so that the registered providers are picked up.
  4. Test the connection from SSMS or the SSIS designer using the TNS alias before the install meeting.
Coordinate with your Oracle DBA

Your Oracle DBA or JDE administrator should provide the TNS alias contents (host, port, and service name) and the read-only credentials RapidReconciler will use. Read-only access to the JDE schemas is sufficient — the package does not write back to JDE.

AS/400 (I-Series) — Driver Guidance

SSIS connects to DB2 for i through the IBMDA400 OLE DB provider, delivered with IBM i Access (formerly Client Access). IBM i Access also installs an ODBC driver, which is a supported alternative if OLE DB is not viable in your environment.

What to Install

  • IBM i AccessInstall IBM i Access Client Solutions (or the older IBM i Access for Windows / Client Access, depending on what your IBM environment supports). This is the package that ships the OLE DB provider and ODBC driver.
  • OLE DB ComponentDuring the IBM i Access install, the OLE DB provider component is not selected by default. Use the custom/feature selection screen to explicitly include it, or the IBMDA400 provider will be missing afterward.
  • ArchitectureInstall both 32-bit and 64-bit versions on the SSIS server. The 32-bit driver is required at SSIS design time in Visual Studio; the 64-bit driver is required for runtime execution under SQL Agent.
  • Provider NameIn SSIS connection managers the OLE DB provider appears as IBM DB2 UDB for iSeries IBMDA400 OLE DB Provider. The ODBC driver appears as IBM i Access ODBC Driver.

Configuration After Install

  1. Confirm the SSIS server can reach the IBM i system over the network — the standard IBM i ports must be open through any firewall between the two systems.
  2. If using ODBC, create a system DSN that points at your IBM i system. Use the same DSN name on both 32-bit and 64-bit ODBC Administrator consoles.
  3. Set the default library or collection to the JDE production library (for example, PRODDTA) on the connection so unqualified table references resolve correctly.
  4. Test the connection using IBM i Access System i Navigator before configuring SSIS.
Licensing reminder

IBM i Access Client Solutions requires a valid IBM entitlement. Confirm with your IBM administrator that the SSIS server is covered under your existing IBM i licensing before installing.

Microsoft SQL Server — Driver Guidance

If your JDE database runs on Microsoft SQL Server, no additional driver software needs to be installed. The OLE DB provider used by SSIS is the same one already installed on every SQL Server host — either the legacy SQLNCLI (SQL Server Native Client) or the modern MSOLEDBSQL (Microsoft OLE DB Driver for SQL Server). Both are supported.

What to Install

  • Driver SoftwareNone required. The SQL Server OLE DB provider is delivered with SQL Server and SQL Server Management Studio. It is already present on the SSIS server.
  • Recommended ProviderUse the modern MSOLEDBSQL driver where available (SQL Server 2019 and later install it by default). It supersedes the legacy SQL Server Native Client and receives current security updates from Microsoft.
  • Optional UpdateIf the SSIS server is on an older SQL Server build and does not have MSOLEDBSQL, download the latest Microsoft OLE DB Driver for SQL Server from Microsoft and install it. This is optional — the legacy Native Client is still supported.

Configuration After Install

  1. Confirm the SSIS server can reach the JDE SQL Server over TCP 1433 (or your custom SQL port) through any firewalls.
  2. Confirm whether your environment uses Windows authentication or SQL authentication, and gather the appropriate credentials.
  3. Test the connection to the JDE SQL Server from SSMS on the SSIS server before the install meeting.
Simplest path

Of the three source platforms, SQL Server is the simplest to set up. Beyond verifying network connectivity and credentials, no additional driver work is needed before the install meeting.

Where the Driver Gets Installed

Regardless of which source platform you chose, the driver software is installed on whichever server runs SSIS:

TopologyWhere to Install Source Drivers
Configuration 1 (Co-located)The single dedicated RapidReconciler server.
Configuration 2 (Separate DB & App)The database server, which also hosts SSIS.
Configuration 3 (Separate SSIS)The dedicated SSIS server only — not the database server, and not the application server.
Why both 32-bit and 64-bit?

This applies to Oracle and AS/400 only. SQL Server Integration Services packages are designed in Visual Studio, which historically runs as a 32-bit process — so the 32-bit driver is required for design and debugging. SSIS executes packages in 64-bit mode by default at runtime — so the 64-bit driver is required for production. Installing both up front avoids reinstalling under pressure later.

Windows Security & Network Rules

Operating-system and network-level prerequisites that should be in place before the install meeting. The Baseline requirements apply to every topology; additional requirements are listed for each configuration.

Drafted from Industry Practice

The recommendations below reflect standard SQL Server and Integration Services security practices. If your organization has stricter internal standards (for example, around service account password policies or firewall change procedures), follow your standards in addition to what is listed here.

Baseline All Topologies

  • Local AdministratorLocal administrative credentials are required on every server during the install meeting. The credentials may be revoked or rotated after install.
  • UAC / ElevationUser Account Control is supported. The install meeting expects the SQL Server, SSIS, and Visual Studio sessions to be launched with elevation (Run as administrator).
  • SQL Server Service AccountA dedicated AD or local service account is recommended for the SQL Server service. Set the password not to expire, or document the rotation procedure with GSI.
  • Antivirus ExclusionsPer Microsoft guidance, exclude SQL Server data and log files (.mdf, .ldf, .ndf) and the SQL Server backup folder from real-time antivirus scanning. Without exclusions, performance degrades and SQL Server can fail under load.
  • Outbound to GSIAllow outbound HTTPS (port 443) from the application server to the GSI hosts listed in the Application Server section. The connection is outbound only; no inbound access from GSI is required.

Configuration 1 Additions Co-located

Single dedicated server hosting database, SSIS, and application services.

  • Service AccountsBecause SQL Server, SSIS, and the application all run on the same machine, a single service account (or the default NT Service\MSSQLSERVER) is sufficient for SQL and SSIS. The application's Windows service can use Local System or a dedicated local account.
  • Windows FirewallNo SQL Server inbound rule is required from other servers, since all components share the same machine. Inbound traffic from the user network to the application's HTTP port should be allowed.
  • AntivirusIn addition to SQL data file exclusions, exclude the RapidReconciler application install directory from real-time scanning.

Configuration 2 Additions Separate DB & App

Database + SSIS on one server, application services on a second server.

  • DB ↔ App AuthenticationThe application server connects to the database server using SQL authentication via the rruser login created during install. No domain trust is required between the two servers, but if your organization prefers Windows authentication, a domain service account in the db_owner role on the RapidReconciler database can be substituted.
  • DB Server FirewallAllow inbound TCP 1433 (or your custom SQL port) on the database server, scoped to the application server's static IP. Block all other sources to minimize exposure.
  • App Server FirewallAllow inbound HTTPS from the user network on the RapidReconciler application port. Allow outbound 1433 to the database server.
  • AntivirusExclude SQL data files on the database server. On the application server, exclude the RapidReconciler application install directory.
  • Time SyncBoth servers should sync to the same time source (domain controller or NTP). Skewed clocks between servers cause intermittent authentication failures.

Configuration 3 Additions Separate SSIS

Database, SSIS, and application services on three separate servers. This configuration adds a second cross-machine boundary that requires careful service-account planning.

  • SSIS ↔ DB AuthenticationThe SSIS server needs the ability to deploy and execute packages against the database server. Windows authentication is preferred — use a domain service account that has appropriate rights on the SQL Server instance. SQL authentication is also supported if Windows auth is not available.
  • SSIS ↔ JDE SourceThe SSIS server reads from the JDE source. Install the OLE DB provider for your JDE platform (AS/400, Oracle, or SQL Server) on the SSIS server, not the database server. Service account or SQL credentials for the JDE source must be available to the SSIS execution context.
  • DB Server FirewallAllow inbound TCP 1433 (or custom SQL port) from both the SSIS server's static IP and the application server's static IP. Block all other sources.
  • SSIS Server FirewallAllow outbound 1433 to the database server. Allow outbound to your JDE source on whatever port and protocol the source requires.
  • Network LatencySSIS reads/writes are latency-sensitive. Recommend gigabit LAN between SSIS server and database server. WAN-separated configurations are not supported.
  • AntivirusExclude SQL data files on the database server. On the SSIS server, exclude the SSIS package folder and execution working directory. On the application server, exclude the RapidReconciler install directory.
  • Time SyncAll three servers must sync to the same time source. Cross-machine Windows authentication fails when clocks differ by more than 5 minutes.
