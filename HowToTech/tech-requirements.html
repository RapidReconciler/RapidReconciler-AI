About RapidReconciler

What RapidReconciler is, what it does, and where it sits in your environment.

What It Does

RapidReconciler is an inventory reconciliation application for organizations running JD Edwards. It compares perpetual inventory balances, end-of-day positions, and cardex transactions against the underlying JD Edwards data, surfaces variances by company and document type, and supports the period-end close by generating suggested journal entries for those variances.

The application is delivered as a web application that customer users access through a browser. It reads from JD Edwards on a scheduled basis, stores its working data in a dedicated SQL Server database, and presents reconciliation views to end users without requiring them to query JD Edwards directly.

How It Integrates

RapidReconciler is a read-mostly companion to JD Edwards. It does not write back to JDE. The application reads from your JDE source on a schedule, transforms the data into a reconciliation-friendly model, and stores the results in its own SQL Server database. Customer users interact only with the RapidReconciler web interface.

ComponentRole
JD Edwards sourceCustomer's existing JDE system. RapidReconciler reads inventory and transaction tables from here on a schedule.
SQL Server Integration Services (SSIS)Extracts data from JDE and loads it into the RapidReconciler database. Runs on a schedule defined during install.
RapidReconciler databaseDedicated SQL Server database that holds the reconciliation working set, configuration, and user data.
RapidReconciler applicationWeb application that customer users access through a browser. Reads from the RR database to render reconciliation views and reports.

What's Included with the Application

  • The RapidReconciler web application itself, licensed by GSI on a per-company basis
  • The SSIS packages that extract data from JD Edwards into the RR database
  • Database schema, stored procedures, and seed data delivered during installation
  • Installation, configuration, and post-install support from GSI

What's Not Included

RapidReconciler does not include the underlying infrastructure software it runs on. The customer is responsible for providing servers, operating systems, SQL Server licensing, and source platform connectivity software. See Software & Licensing for the full breakdown.

Deployment Architecture

RapidReconciler can be deployed in three supported topologies. The hardware specifications for each server role are the same across all three; the difference is how the components are distributed across machines.

The Three Topologies

Configuration 3
Separate SSIS
Specialized
Database, SSIS, and application services on three separate servers.

Most customers choose Configuration 2

For most enterprise deployments, Configuration 2 is the right balance — separation of database and application services without the operational overhead of a dedicated SSIS host. Configuration 1 fits small or pilot deployments where cost of provisioning a second server isn't justified. Configuration 3 is appropriate only when there is a specific organizational reason to isolate SSIS.

JD Edwards Source Platform

RapidReconciler reads from your JD Edwards database on a schedule. Three source platforms are supported. Each requires specific connectivity software on the host that runs SSIS.

Supported Platforms

PlatformConnectivity RequiredCustomer Provides
Oracle Oracle Provider for OLE DB (delivered with the Oracle Database Client or ODAC) Oracle client software, both 32-bit and 64-bit, in a shared Oracle home
AS/400 (I-Series) IBMDA400 OLE DB provider, delivered with IBM i Access IBM i Access Client Solutions, both 32-bit and 64-bit, with the OLE DB component explicitly enabled

What RapidReconciler Needs from the Source

  • Read-only access to the JDE schemas containing inventory and transaction data
  • Stable connectivity from the SSIS host to the JDE source — gigabit LAN preferred
  • A service account or SQL credentials that can be used by the SSIS execution context
  • Confirmation of four data dictionary settings (decimal precision for cost and quantity fields) — these are captured during installation prep

RapidReconciler does not write back to JD Edwards. Read-only credentials are sufficient.

Why Both 32-Bit and 64-Bit Drivers

This applies only to Oracle and AS/400. SQL Server Integration Services packages are designed in Visual Studio, which historically runs as a 32-bit process — so the 32-bit driver is required at design time. SSIS executes packages in 64-bit mode by default at runtime — so the 64-bit driver is required for production. Installing both up front avoids reinstalling under pressure later.

Licensing Notes

  • IBM i Access requires a valid IBM entitlement. Confirm with your IBM administrator that the SSIS host is covered before planning installation.
  • Oracle Database Client is generally available without separate licensing for clients connecting to a licensed Oracle Database, but confirm with your Oracle DBA or licensing contact.
  • Microsoft SQL Server connectivity is delivered with SQL Server itself and requires no additional licensing.
Confirm with your JDE administrator

Your JD Edwards administrator should be able to tell you which of the three platforms hosts your production JDE database. The platform choice affects only the connectivity software to install — the RapidReconciler application itself is the same regardless of source.

Time Synchronization

All RR servers should synchronize time to the same source — typically your domain controller or a corporate NTP server. Cross-machine Windows authentication fails when clocks differ by more than five minutes, and intermittent authentication failures of this kind are time-consuming to diagnose during install.

Network Latency

SSIS data flows are latency-sensitive. Gigabit LAN is required between the SSIS host and the database server in Configuration 3. WAN-separated configurations are not supported.

Security Posture

RapidReconciler is designed to operate within standard enterprise security practices. This section summarizes what your security team will need to plan for.

Account Model

RapidReconciler uses a small set of well-defined accounts. The exact configuration depends on the chosen topology, but the model is consistent.

  • SQL Server Service AccountA dedicated AD or local service account is recommended for the SQL Server service. Set the password not to expire, or document the rotation procedure with GSI.
  • Application Database UserA SQL-based login (rruser) is created during install for the application's connection to the database. Windows authentication via a domain service account in the db_owner role on the RapidReconciler database can be substituted if your organization prefers it.
  • Source Read AccountA read-only account on the JDE source database, used by SSIS to extract data. Provided by the customer's JDE administrator.
  • Local AdministratorLocal administrative credentials on each RR server are required during the install meeting. These can be revoked or rotated after install.

Antivirus Considerations

Per Microsoft guidance, real-time antivirus scanning of SQL Server data files degrades performance and can cause SQL Server to fail under load. Plan for the following exclusions on each affected server:

  • SQL Server data and log files (.mdf, .ldf, .ndf) on the database server
  • The SQL Server backup folder on the database server
  • The RapidReconciler application install directory on the application server
  • The SSIS package folder and execution working directory on whichever host runs SSIS

Firewall Posture

Firewall rules should follow the principle of least privilege — open only the ports needed, scoped to the specific source IPs that need them.

  • Database server inbound 1433 should be scoped to the application server's static IP (and the SSIS server's static IP in Configuration 3). Block all other sources.
  • Application server inbound HTTPS should be scoped to the user network range that will access RapidReconciler.
  • The outbound connection from the application server to GSI hosts uses port 443 and is outbound-only — no inbound exception is required.

Authentication Between Servers

For Configurations 2 and 3, the application server and (in Configuration 3) the SSIS server need to authenticate to the database server. SQL authentication via the rruser login is the default and requires no domain trust between machines. Windows authentication via a domain service account is supported as an alternative and is preferred if your security policy mandates it.

Conformance with internal standards

The recommendations above reflect standard Microsoft and industry guidance for SQL Server and Integration Services deployments. If your organization has stricter internal standards — for example around password policies, firewall change procedures, or service-account governance — apply your standards in addition to what's listed here. GSI will work within your existing security framework during installation.
