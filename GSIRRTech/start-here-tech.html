Who This Is For

You’re a new GSI tech-team member who will be installing RapidReconciler at customer sites, performing server migrations, and supporting the customers afterward. You’re technically capable — you have SQL, Windows-server, and basic networking comfort — but you haven’t yet seen RapidReconciler from the GSI side.

This guide assumes you’ve read enough of the role to know that RapidReconciler is a JDE reconciliation product, but you don’t need any prior knowledge of VALC, the agent architecture, or the install procedure. What you’ll get here is the “you are here” map — what each tool does, where each procedure lives, and what to read in what order.

What VALC Is

Five-minute orientation to the central tool you’ll use every working day.

The 30-Second Version

VALC is GSI’s internal admin console for every RapidReconciler customer. It’s where customers get added, configured, licensed, troubleshot, and decommissioned. Tech and Customer Service share this tool; the customer themselves never sees it.

Functionally, it does four things:

  • Customer registry — name, contract dates, modules licensed, contact info.
  • Agent control plane — tracks each customer’s RapidReconciler Agent status, IPs, and version. The agent reports back; VALC tells the agent what database(s) to talk to.
  • Database registration — a customer’s SQL Server databases are registered here so the agent knows which ones to monitor.
  • Lifecycle gates — the Setup Step field on each customer record drives what they can and can’t do (e.g. agent install download isn’t offered until Setup Step = Created).

The Five Sidebar Items

After login, you see five items in the left sidebar. They’re not equally important.

!

Common new-tech mistake. Poking around in SQL Scripts thinking it’s part of the install workflow. It’s not. Knowing which tools to ignore is half of working efficiently.

What VALC Is Not

  • Not the customer-facing application. Customers log into https://rapidreconciler.getgsi.com for the actual product. VALC is GSI staff only.
  • Not where you write SQL queries against customer data. VALC controls the agent that talks to the customer’s database, but you don’t query their data through VALC.
  • Not where customers manage their own users. Each customer’s administrator manages their users inside the RapidReconciler application itself, not VALC.

Your First Session

A read-only tour of every screen you’ll touch in your daily work. The goal isn’t to learn how to do any of these things yet — just to recognize them when they appear in the install reference.

Setup before you begin

Open VALC in a separate browser window or tab. Sit it side-by-side with this document if you have screen space. Pick an existing customer with a healthy install (Status badge says Connected) to look at as you go — ask a senior tech for a recommendation.

You will not create, edit, or delete anything during this tour. When you open popups like Add Client or Edit Database, you’ll close them without saving. The goal is recognition, not action.

The Walk — 6 Screens, ~30 Minutes

Step 1 — Read the Clients Grid

Click Clients in the left sidebar. You’re now looking at a list of every customer GSI has set up.

Read the column headers from left to right. Two things to notice:

!

Two columns are called “status” — they mean different things. The leftmost Status column is whether the agent is connected to VALC right now (green Connected / red Disconnected). The System Status column further right is whether the most recent data sync from JDE succeeded.

These two badges fail in different ways and require different troubleshooting paths. Mixing them up sends you down the wrong rabbit hole.

At the top-right of the grid, find:

  • Actives dropdown — filters which customers you see. Other values surface inactive or archived clients.
  • Create Client button (blue) — how new customers get added (you’ll preview that form in Step 5).

Step 2 — Hover the Options Column

Find any row and hover on the icons in the rightmost Options column. You should see two icons: Manage Clients and Manage Deploys.

Manage Clients is by far the most-used button in the whole app for tech — it’s where you go to do anything for that specific customer.

Step 3 — Open Manage Clients

Pick a healthy active customer (Status = Connected). Click their Manage Clients icon. You’ll be taken to their detail screen with four tabs across the top: Client Details, Databases, Companies, User Accounts. You land on Client Details by default.

Look at the upper portion: Client Name, Domain URL, Setup Step, Agent IPs, License dates, Version. The Setup Step field is the most important one to know about — it tracks where the customer is in their lifecycle (Created, Connected, etc.) and the agent install requires it to be on Created before the download prompt appears.

Find the toggles on the right:

  • Active — whether this client record is currently in service.
  • Password Policy — when ON, that customer’s users are subject to the complex password rules.

Below the toggles, find the modules section. Each module has its own Active checkbox. These are licensed per the customer’s contract — only check the ones they’ve paid for.

Locate Agent Internal IP Address and Agent External IP Address. Both are the customer’s application server IPs (where the RR Agent runs). The database server IP is recorded separately on the Databases tab — even if it’s the same physical machine. This split trips up new techs constantly.

Step 4 — Tabs 2, 3, 4 (Quick Pass)

Click each remaining tab in turn:

For Databases, watch out for status states other than Online:

  • Online (green) — healthy.
  • Unknown — agent reached the database before but has lost contact.
  • Stopped — database has never connected since being registered.

You don’t need to learn the diagnostics today — just recognize the states.

For User Accounts: this is where the initial GSI Admin user is created during install. Subsequent users are usually added by the customer’s own administrator inside the RR application, not here.

Step 5 — Open Add Client (Read-Only)

Go back to the Clients grid (sidebar → Clients). Click the blue Create Client button at the top-right. The Add Client popup appears.

Click around to recognize the fields. Click X to close. Do not click Confirm. Real Add Client work happens in Part 1 of the install reference.

Step 6 — Open Edit Database (Read-Only)

Go back to the customer’s Manage Client screen, click the Databases tab, and on any row click the edit icon in the Options column. The Edit Database popup appears.

Three things to notice while you have it open:

  • Address is the database server’s IP — not the application server’s, even when they’re the same machine.
  • Server Port (Don’t change) is an internal port — different from the SQL port the customer’s DBA might mention.
  • Max Memory options (2/4/6/8/12 GB) controls RAM on the application server, not the database. Default is 2 GB; bump only if performance warrants.

Click X to close. Do not click Confirm.

Try this now

The 6-step walk above is meant to be done, not just read. Open VALC and work through the six steps in order. Don’t aim for understanding everything you see — aim for being able to find each landmark.

If you got stuck on any step, that’s the part to ask your reviewer about before moving on to next week’s reading.

  • After your first session
  • Log into VALC and reach the Clients grid.
  • Identify the difference between the leftmost Status column and the System Status column.
  • Open a customer’s Manage Clients view and name what each of the four tabs is for.

Your First Real Install

You’ve read the docs and walked through VALC. Time to do one for real. The pattern is shadow first, co-lead second, lead solo third — with the install reference open the entire time.

Two Recommendations Before Your First Install

These aren’t enforced — both noticeably reduce first-install stress.

!

Shadow before you co-lead. Sit in on a senior tech’s install end-to-end before you’re responsible for any of the steps. Watch how they handle the discovery call, the live work, the troubleshooting moment that always shows up at least once. Take notes on what isn’t in the docs — the “the way we actually do this” details.

!

Pre-stage your environment. Before the customer joins the working session, have these open in your browser tabs: VALC (logged in), the install reference doc, the production-database guide if it’s in scope, and a notepad for IPs and credentials. Walk through Part 1 yourself before customer IT joins so you’re not learning the screens with them watching.

Pre-Install Checklist

  • Before the working session
  • Discovery call completed; topology and access requirements documented.
  • Customer IT contact identified and scheduled for the working session.
  • Network access requirements sent to customer IT and confirmed.
  • VALC client record created (Setup Step = Created).
  • Initial GSI Admin user created on the new client.
  • Modules configured per the contract.
  • Cloudflare DNS request submitted to Daren; A record confirmed live.
  • Install reference doc and production-database guide bookmarked.

Top Three New-Tech Mistakes Specific to Installs

1
Mistake 1 of 3
Confusing the application IP with the database IP

Both are recorded in VALC, but in different places. The agent IPs (Internal and External) on Client Details are the application server. The Address field in Edit Database is the database server. Even when they’re the same physical machine, the values are recorded separately.

Why it matters: putting the wrong IP in Cloudflare or in the database registration is a top-3 cause of “the agent installed but data isn’t flowing” tickets.

Cross-reference: Troubleshooting → Cloudflare Update Made but Data Not Appearing
2
Mistake 2 of 3
Forgetting to set Setup Step = Created before the agent install

The agent download prompt won’t appear at rapidreconciler.getgsi.com if the customer’s Setup Step is anything other than Created. New techs sit on the customer’s call confused for fifteen minutes about “why isn’t the download showing up” before discovering this.

Always verify Setup Step = Created on the Client Details tab right before the customer kicks off the agent install. Twenty-second check.

Cross-reference: Troubleshooting → Setup Step Not Advancing
3
Mistake 3 of 3
Licensing the wrong company numbers

The Companies tab shows every JDE company the customer’s database has, but only the ones in the contract should be licensed. Over-licensing creates a billing/compliance problem; under-licensing means the customer can’t see data they’re paying for.

Before checking any boxes, get the customer’s contract or sales contact to confirm the exact list. If you’re unsure, leave them all unchecked and resolve before customer handoff — never guess.

If Something Goes Wrong During the Working Session

Something usually does. The pattern that works:

  1. Stay on the call. The customer’s IT person is your most valuable resource — they have access to the server you don’t.
  2. Open the agent log file. The install reference’s Agent Log Files section tells you where it lives. The log will tell you more than any error message.
  3. Match the symptom to a Troubleshooting subsection. The Troubleshooting topic has the six most common patterns. If your symptom matches, follow that procedure.
  4. Don’t escalate prematurely. If you’ve walked through the matching troubleshooting section and it didn’t resolve, that’s when to bring in a senior tech — not before.
  5. Document everything. Even if the issue resolves quickly, capture what happened and what fixed it. The Common Requests doc has gaps that get filled in this way.
i

The install reference is your active companion during a real install. Have it open. Don’t try to do an install from memory. Senior techs follow it section-by-section too — the difference is they know which sections to check before they need to.

Vocabulary

RapidReconciler-specific terms that aren’t standard SQL/Windows/networking vocabulary. Reach for this anytime — it’s the doc you’ll consult quickly during your first weeks and almost never after.

Agent (RR Agent)
The Windows service installed on the customer’s application server that connects out to VALC, reads from the customer’s JDE/SQL Server database(s), and ships the data needed for reconciliation. Customers download it from rapidreconciler.getgsi.com after logging in. Each customer has exactly one agent (per server).
Cloudflare
GSI’s DNS layer. Each customer’s customer.getgsi.com A record is managed there. Cloudflare changes go through Daren — tech submits the request, Daren actions it. Troubleshooting → Cloudflare Update Made but Data Not Appearing.
Companies (Companies tab)
JDE company numbers licensed for a customer. The list is populated automatically from the customer’s JDE; tech checks the boxes for the companies the customer paid for. Over-licensing is a compliance issue; under-licensing means missing data in the customer’s UI. Part 3 Step 3: License Company Numbers.
Discovery Call
The pre-install conversation with the customer’s IT contact. Topology, network access, JDE database details. Drives where the install lands and which scripts you’ll need. Overview → Discovery Call.
GSI Admin user
The initial user account created during install on each customer’s record (User Accounts tab in VALC). Not the same as a customer’s administrator account — this is GSI’s back-door login that customers can’t see, used for customer-side troubleshooting and for customer admins to reset their own access if locked out.
Modules
Optional functional areas (Inventory, In Transit, PO Receipts, etc.) within RapidReconciler. Each is licensed separately per the customer’s contract. Toggled on Client Details. Only enable what they’ve paid for.
Password Policy (toggle)
The optional complex-password requirements for a customer’s users. When ON, users are subject to length, complexity, history, and expiry rules. Toggled on Client Details. Customers request activation via support email. Password Policy.
Setup Step
The lifecycle field on Client Details that controls what the customer can do at any moment. Most relevant value: Created — the agent download prompt only appears at rapidreconciler.getgsi.com when this is set. New techs forget to set/verify this and lose 15 minutes of an install call before realizing why.
SSIS Package / SSISDB
SQL Server Integration Services artifact that copies data from the customer’s JDE database into RapidReconciler_Prod. Lives in the SSISDB catalog inside SQL Server. Created and deployed during a database install (production database guide), not the agent install. Installing the Production Database → Tour 3: Configure & Deploy SSIS Package.
System Status
The column on the Clients grid (and field on Client Details) showing whether the most recent JDE-to-RR data sync succeeded. Different from the leftmost Status column, which shows agent connectivity. The two indicators fail in different ways and require different fixes.
VALC
GSI’s internal admin console for all RapidReconciler customers. Where customers are added, configured, licensed, and supported. Tech and Customer Service share it; customers never see it. Not to be confused with the customer-facing RR application at rapidreconciler.getgsi.com. Installing a Client in VALC → About VALC.
Working Session
The 2–3 hour scheduled web meeting where GSI tech and customer IT do Part 2 of the install live together. Calendar-scheduled, not improvised. Customer IT needs admin access to the application server during this window.
