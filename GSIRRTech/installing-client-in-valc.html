Overview & Prerequisites

About the install process, the discovery call, and the information to collect from the customer before kicking off Part 1.

This section covers what you need to know and gather before opening VALC. Don't skip it — missing information here causes rework later in the install.

About the Process

Setting up a new customer in VALC and installing the RapidReconciler Agent on their application server is a three-phase process:

  • Part 1: Initial VALC Setup — Creating the client record, initial user account, and module configuration in the VALC portal.
  • Part 2: RapidReconciler Agent Installation — Downloading, installing, and validating the agent on the customer's server.
  • Part 3: Completing the Setup in VALC — Verifying connectivity, database status, and licensing company numbers.

Agent Deployment Constraints

Two architectural rules that shape every install and migration:

  • One agent per customer network. Only a single instance of the RapidReconciler Agent should be installed on the customer's internal network. Multiple agents pointing at the same VALC client record will compete and produce inconsistent state.
  • One agent supports multiple databases. A single agent can serve any number of RapidReconciler databases (Prod, Test, Dev, Train) — including databases hosted on different SQL Servers. Each database is registered separately in VALC (see Part 3, Step 1) but they all communicate through the same agent.

What the RR Agent Transmits to GSI

The agent maintains an outbound-only connection from the customer's application server to GSI's VALC infrastructure (256-bit encrypted; no inbound traffic to the customer's network). It transmits the following data, used to keep VALC's view of the customer environment current and to support licensing and version control:

DataPurpose
Public-facing IP of the application serverPopulates Agent External IP Address on Client Details; used for connectivity verification.
Internal IP of the application serverPopulates Agent Internal IP Address on Client Details.
IP address and port of the database serverSurfaced on the Databases tab; used to confirm database reachability.
Credentials used by the agent to access the RapidReconciler databaseStored in VALC so the agent can be re-pointed or reconfigured without re-prompting the customer. GSI cannot retrieve a password — only reset it.
RapidReconciler database name(s) and port number(s)One row per registered database on the Databases tab.
RapidReconciler usernames and passwordsStored hashed; GSI cannot retrieve them, only reset.
User access security settings (companies and tabs each user can see)Allows GSI support to assist with access questions without needing screen-share into the customer environment.
5-digit company number and name from JD EdwardsUsed solely for licensing — the company number drives the Companies tab in VALC and enforces the license count from the contract.

Knowing what the agent reports back is useful when answering customer security questions, when explaining why a value appears in VALC without being entered there manually, and when triaging mismatches between the customer's environment and what VALC shows.

Discovery Call

A discovery call is normally held with the customer's IT contact before installation begins. The goal is to confirm everything needed to schedule and execute the install without back-and-forth during the work itself. Use the checklist below as a guide.

Server Topology

Confirm whether the customer is deploying RapidReconciler on a single dedicated server (application and database co-located) or on separate application and database servers. This determines where each IP address is recorded in VALC:

  • The application server IP information is captured on the Client Details tab (Agent Internal IP Address, Agent External IP Address).
  • The database server IP and port are captured on the Databases tab, on each database record.

If the customer is using a single dedicated server, the same IP appears in both places. The same topology question also drives the procedure used during a future Server Migration.

VALC (Version and Licensing Control) is a GSI-developed web application hosted on Microsoft Azure. It is used to manage RapidReconciler and Genius customers. All new customers must be added to VALC before the installation process can begin.

Part 3: Completing the Setup in VALC

Register the database, verify it goes Online, license company numbers, and complete the customer handoff.

With the agent installed and connectivity confirmed, this final phase points VALC at the customer's database, verifies it's reachable, licenses the customer's company numbers per the purchase agreement, and hands the environment off to the customer's RapidReconciler administrator.

Step 1 — Register the Database in VALC

The customer's RapidReconciler_Prod database already exists on their SQL Server (created as part of the pre-install database setup). This step tells VALC how to reach it: address, credentials, job name, and so on. The Edit Database popup is also reused to register Test, Dev, or Training databases when those are part of the contract, and to update connection details later in the lifecycle.

Where the IP Goes

The Address field is the IP of the customer's database server. If the customer is using a single dedicated server, this is the same IP as the application server (recorded on the Client Details tab). If they are using separate servers, the database IP is recorded only here. Confirm topology with the discovery-call notes from the Overview.

Adding a Second or Third Database

When a customer adds a Test, Dev, or Training database, do not run the database creation scripts again — those scripts are coded with RapidReconciler_Prod references and will fail or produce inconsistent objects. Instead, on the customer's SQL Server: back up the existing RapidReconciler_Prod database and restore it under the new name (RapidReconciler_Test, RapidReconciler_Dev, etc.). Then register the new database in VALC using the procedure below. This ensures schema and seed data are consistent across environments.

  1. From the Clients grid, click Manage Clients for the client.
  2. Select the Databases tab.
  3. Click Create Database (bottom-left of the grid) for a new record, or click the edit icon on an existing row to update one. Either action opens the Edit Database popup.
  4. Complete the form using the field reference below, then click Confirm.
FieldValue

Step 4 — Customer Handoff

With licensing complete, the install is functionally done. The remaining step is to formally hand the environment off to the customer's RapidReconciler administrator so they can begin setting up their own users and using the application.

What to Send the Customer

Send the customer's IT contact and RapidReconciler administrator a handoff email containing:

  • Login URL — the customer's .getgsi.com subdomain (from the Domain URL field on the Client Details tab).
  • Initial GSI Admin credentials — the email and temporary password for the administrator user created in Part 1, Step 2. The administrator will be prompted to reset the password on first login.
  • Pointer to user management — a brief note that the administrator can add additional users from the User Accounts tab in their environment.
  • Password policy status — whether the complex password policy was enabled at install. If not, mention they can request activation later via rrsupport@getgsi.com. See the Password Policy reference for full details.
  • Support contact — rrsupport@getgsi.com for ongoing questions.
Suggested Subject Line

RapidReconciler — Your environment is ready or RapidReconciler install complete — login details inside

Confirm with the Customer

After sending the handoff email, confirm the customer has successfully logged in at least once. The simplest confirmation is the customer reporting back that they reached the application; the administrative confirmation is to check the User Accounts tab in VALC for that client and verify the GSI Admin user shows recent activity.

What's Now the Customer's Responsibility

Once the handoff is complete, the customer's RapidReconciler administrator owns:

  • Adding and removing user accounts for their team
  • Resetting passwords for users who get locked out (see Password Policy for the procedure)
  • Day-to-day use of the application

GSI continues to own the underlying environment — VALC configuration, agent and database health, server-side updates, and the kinds of changes captured in Common Requests.

Troubleshooting

Six diagnostic patterns for the most common install problems, with the agent log file as the primary diagnostic tool.

This section covers the most common failure modes encountered during a client install and in production. Each entry lists the symptom, the most likely causes, and the steps to remediate. If a problem isn't covered here or remediation steps don't resolve it, escalate to the GSI database administrator or a senior tech.

Diagnostic Tools

Agent Log Files

The RapidReconciler Agent writes detailed log files on the customer's application server. These are the primary source of truth for diagnosing connectivity issues — when a scenario below says "check the agent logs," this is where to look.

Location: C:\Program Files\Rapid Reconciler\logs\

What's there:

  • rr-valc-agent_YYYYMMDD.out — daily output log. Open today's file (matching the current date) to see recent agent activity. Most connectivity diagnostics live here.
  • rr-valc-agent_YYYYMMDD.err — daily error log. Inspect alongside the .out file when the agent is reporting issues.
  • rr-valc-agent.wrapper — service wrapper log. Useful when the agent service itself isn't starting.

Double-click the .out file to open it in the default text viewer. Scroll to the bottom for the most recent entries.

Tip

If you're diagnosing remotely, ask the customer's IT contact to send you the latest .out file rather than transcribing errors over a call. The full log usually contains more context than any single error message they'll quote.

Database Status: Unknown or Stopped

Symptom: On the Databases tab, a row shows Unknown (orange) or Stopped (orange) instead of Online (green). See Part 3, Step 2 for an example image.

Tell them apart:

  • Stopped usually means the database has never connected since being registered in VALC. The Revision Number column is typically blank — no revision has been reported back yet.
  • Unknown usually means the agent had previously reached the database (so a Revision Number is populated) but has since lost contact.

Likely causes:

  • Connection details on the Edit Database popup are wrong — IP, port, username, password, or name don't match what's actually on the customer's SQL Server
  • RR Agent service on the customer's application server is stopped or has lost network access
  • Firewall on the database server is blocking the agent's connection on the configured port
  • The physical database doesn't exist on SQL Server (registration in VALC succeeded but the underlying database wasn't created or has been removed)
  • SQL Server Authentication is disabled or the rruser login was not created

Remediation:

  1. Open the Edit Database popup for the failing row and verify every field matches the actual database — see Part 3, Step 1 for the field reference.
  2. Confirm the RR Agent service is running on the customer's application server. If not, start it.
  3. Check today's agent .out log file (see Diagnostic Tools above) for connection errors or stack traces.
  4. From the application server, attempt a manual connection to the database server on the configured port (e.g., telnet 10.x.x.x 1433) to confirm reachability.
  5. Confirm the customer's DBA has created the RapidReconciler_Prod database and the rruser SQL Authentication login.
  6. If everything looks correct and status remains failed, escalate to the GSI database administrator.

Companies Tab is Empty

Symptom: Database shows Online on the Databases tab, but selecting it from the dropdown on the Companies tab shows no company numbers to license.

Likely causes:

  • The first sync from JDE has not yet completed — Online status confirms VALC can reach the database, but company numbers are populated from JDE through the SSIS package's first run
  • The SQL Agent job is not running or is disabled on the database server
  • The SSIS package's source connection (to JDE) is misconfigured, even though the destination (to RapidReconciler_Prod) is working
  • (verify) The customer's JDE environment has no companies in the source tables that the SSIS package reads

Remediation:

  1. On the Companies tab, check the Last Company Sync timestamp at the top-right. If it shows recent activity, give the sync a few more minutes to complete and refresh the page.
  2. On the database server, verify the RapidReconciler SQL Agent job is enabled and has run successfully at least once.
  3. Check the SQL Agent job history for errors on the most recent run.
  4. If the job is failing or has never run, escalate to the GSI database administrator to investigate the SSIS package and JDE connection.

Agent Install Hangs at "Validating Data"

Symptom: During agent install (Part 2, Step 2), the browser displays Validating Data indefinitely instead of progressing to the SQL Server connection prompt or Deploying.

Likely causes:

  • The customer's application server cannot reach the database server on the configured SQL port
  • SQL credentials provided in the connection prompt are incorrect
  • The RapidReconciler_Prod database is not yet created on SQL Server, or schema is incomplete
  • Firewall on the database server is blocking inbound traffic on port 1433 (or the customer's configured port)
  • SQL Server Browser service is not running on the database server (verify)

Remediation:

  1. From the application server (where the agent is being installed), attempt a manual connection to the database server on the configured port to verify reachability.
  2. Confirm the database, the rruser login, and the SQL Authentication mode are all in place per the pre-install database setup.
  3. If the prompt is reached but still fails, double-check the values being entered match the customer's SQL configuration exactly.
  4. Check Windows Event Viewer on the application server for RapidReconciler-related errors. Also check today's agent .out and .err log files (see Diagnostic Tools above) — the agent logs are usually more specific than the Event Viewer entries.
  5. If the install needs to be restarted, ensure the Setup Step is set back to Created in VALC before retrying — see Server Migration for the rationale.

Clients Grid Row Not Updating to Connected

Symptom: Agent install reported Installation Complete, but on the VALC Clients grid the row's Status badge stays at the previous value (e.g., Disconnected or whatever it was before install) instead of switching to Connected.

Likely causes:

  • DNS for the customer's .getgsi.com subdomain has not yet propagated after Cloudflare update
  • RR Agent and RR Services are installed but one or both services are not running
  • VALC has not refreshed — the underlying agent has connected but the grid row hasn't updated yet
  • Agent External IP on the Client Details tab does not match the customer's actual external IP

Remediation:

  1. Refresh the Clients grid in your browser (the page does not auto-refresh). Wait a few minutes if the install just completed.
  2. From the customer's application server, try resolving the customer's .getgsi.com URL — if DNS still returns the old IP, the Cloudflare update hasn't propagated yet. See Server Migration for the Cloudflare contact and process.
  3. On the application server, confirm RapidReconciler Agent and RapidReconciler Services are both running in the Windows Services console.
  4. Verify the Agent External IP Address on the Client Details tab matches the customer's actual external IP.
  5. If the row still hasn't updated, check today's agent .out log file (see Diagnostic Tools above) for connection errors back to VALC.

Setup Step Not Advancing

Symptom: After completing an install or migration, the Setup Step field on the Client Details tab remains at Created (or an earlier value) and does not advance automatically.

Background: The Setup Step field tracks where a client is in their lifecycle. Created is the state required for the agent to display the install prompt on first login (see Server Migration). After install completes successfully, it should advance — but in some cases the state needs to be moved manually.

Remediation:

  1. Confirm the install actually completed end-to-end: agent installed, database Online, connectivity row showing Connected.
  2. If all of the above are good, manually advance the Setup Step on the Client Details tab to the appropriate next value (verify which value the field expects after install).
  3. If you're not sure what value to set, check another active client for reference.

Cloudflare Update Made but Data Not Appearing

Symptom: Cloudflare has been updated by the GSI network admin and confirmed propagated, but data still doesn't appear in the customer's RapidReconciler UI.

Likely causes:

  • SQL Agent job schedule is not yet enabled (this is the last step in Server Migration and is intentionally deferred until after Cloudflare confirmation)
  • SQL Agent job is enabled but has not yet had a chance to run on its schedule
  • SSIS package connection strings still point to the old server (after a migration)
  • Browser cache or session is showing stale data

Remediation:

  1. On the database server, confirm the SQL Agent job schedule is enabled, then trigger a one-time manual run to validate.
  2. Check the job history for errors.
  3. Have the customer log out and log back in, or hard-refresh their browser.
  4. If the issue is post-migration, double-check SSIS package connection strings on the new server.
Diagnostic Quick-Reference

When a customer reports "RapidReconciler isn't working," the fastest triage path is usually:

  1. Check the customer's row on the VALC Clients grid — Status, Services version, Agent version, System Status.
  2. Open Manage Client > Databases tab — confirm Online for every registered database.
  3. If both look healthy, the issue is likely on the JDE-side data flow (SQL Agent job, SSIS package).
  4. If either looks unhealthy, work backward from the failed signal using the relevant section above.

Password Policy

The complex password policy: requirements, login, reset, administration, and how to flip the per-customer toggle.

RapidReconciler supports an optional complex password policy that can be enabled per client by the GSI tech team. When enabled, all user passwords for that client must conform to the requirements below. The policy is designed to protect sensitive financial reconciliation data and align with common enterprise security standards.

Password Requirements

When the complex password policy is enabled, all user passwords must meet the following criteria:

RequirementDetail
Minimum length8 characters
Name restrictionCannot contain the user's account name or parts of their full name exceeding two consecutive characters
Password historyCannot match any of the last 10 passwords
ExpiryMust be changed every 90 days
Storage & transmissionMust not be displayed, stored, or transmitted in clear text

Character Complexity

Passwords must contain characters from at least three of the following four categories:

  • English uppercase characters (A through Z)
  • English lowercase characters (a through z)
  • Base 10 digits (0 through 9)
  • Non-alphabetic characters (e.g. !, $, #, %)
Tip

A strong password example would be Blue$ky92 — it contains uppercase, lowercase, a digit, and a special character, and does not resemble a name or a previously used password.

Logging In

When a user attempts to log in, they will be directed to the Password Reset screen if either of the following conditions is met:

  1. 90 days have elapsed since their last password change.
  2. They click the "Forgot your password?" link and then follow the reset link sent to their email.
Note

The password reset link sent by email is time-limited. If the link has expired, the user should repeat the "Forgot your password?" process to receive a new link. If the issue persists, contact the customer's RapidReconciler administrator.

Resetting a Password

On the Password Reset screen, the new password must conform to the policy outlined above. The Confirm button remains disabled until the password meets the 8-character minimum length requirement.

Common reasons a new password may be rejected:

  • The password is fewer than 8 characters
  • The password does not meet the three-category complexity requirement
  • The password contains the user's name or account name
  • The password matches one of the last 10 previously used passwords

Administration

Assigning a Temporary Password

The customer's RapidReconciler administrator can assign a temporary password to a new user. On first login, the user will be required to reset the temporary password before gaining access to the application.

Best Practice

When assigning a temporary password, notify the user through a separate communication channel (e.g. phone or a separate email) rather than including the password in the same message as the login instructions. This reduces the risk of unauthorized access.

Enabling or Disabling the Password Policy

The complex password policy is enabled or disabled at the client level via the Password Policy toggle on the Client Details tab in VALC (see Part 1, Step 3). When disabled, users are not subject to the complexity, history, or expiry requirements described above.

Recommendation

It is strongly recommended to keep the complex password policy enabled in all production environments to protect access to sensitive financial data.

Locked or Inaccessible Accounts

If a user is unable to log in and cannot complete the password reset process (e.g. they no longer have access to the email address on file), the customer's RapidReconciler administrator can assign a new temporary password directly through the User Accounts section in VALC.

Requesting Activation

Customers request activation of the complex password policy by emailing GSI support. The GSI tech team enables the Password Policy toggle on the client's Client Details tab once the request is processed.

The customer's email to rrsupport@getgsi.com should include:

  • Company name and RapidReconciler account details
  • The name of the customer's RapidReconciler administrator
  • Requested activation date
Note

It is recommended to notify all users in advance before the policy is enabled. Existing users will be required to reset their passwords at their next login once the policy is active.

Message Users

How to post system-wide messages users see at login — for maintenance windows, version changes, and similar broadcasts.

The Message Users page in VALC broadcasts a short text message to every RapidReconciler user across every client. Users see the message when they next log in to the application. Messages display only between their Begin and End dates, then disappear automatically.

Common uses:

  • Announcing scheduled maintenance windows
  • Notifying users of deploys, upgrades, or feature changes
  • Time-bounded notices (e.g., "Reconciliation jobs delayed Friday morning")

Posting a Message

  1. From the main navigation, click Message Users.
  2. Click New Message at the top-right of the page.
  3. Enter the message text, the Begin date (when the message starts appearing), and the End date (when it stops).
  4. Save the message. It will appear in the table on the Message Users page and will be shown to users who log in during the active window.

Editing or Removing a Message

Each row in the messages table has two icons in the Options column:

  • Edit (pencil) — open the message for editing. Useful for correcting typos or extending the End date if a maintenance window runs long.
  • Delete (trash) — remove the message permanently. Use this to take down a notice early; otherwise messages disappear automatically when the End date passes.
Important

Messages are seen by every RapidReconciler user at every customer. Keep wording professional and customer-neutral — do not include client-specific details, internal jargon, or sensitive information. If a notice only applies to one customer, contact them directly instead of using this broadcast feature.

Best Practices
  • Post maintenance notices at least a few days in advance so users have time to plan around them.
  • Use a clear Begin/End window — end-dating a message removes it without manual cleanup.
  • Keep messages short. Users see them once at login; long messages get dismissed without being read.
  • Avoid stacking multiple overlapping messages — users see all active messages at once, which dilutes attention.

SQL Scripts (Developer Reference)

Developer-only tool for applying SQL update scripts across customer databases. Tech support uses this rarely; included for completeness.

SQL Scripts (Developer Reference)

Intended Audience

This page documents the SQL Scripts page in VALC, which is reserved for RR developers. The GSI tech team responsible for client installs does not normally use this feature. This section is included here as a single self-contained reference for the team that does.

The SQL Scripts page provides a controlled way to apply update scripts (schema changes, data fixes, patches) across customer databases without logging in to each customer's SQL Server directly. A developer selects which databases to target, chooses an update script, and executes — pass/fail results are reported back inline.

Page Layout

The page is divided into three working areas:

AreaPurpose
Database list (left)Lists all databases registered across all customer clients in VALC, formatted as {ClientName} - {DatabaseName} - {RevisionNumber}. Each row has a checkbox for selection.
Filters (top-left)Database Selection filters the list by the database's Category (Production, Test, Archive, Change Control, Inactive — see Part 3, Step 1 for where Category is set). Revision Number is a freeform text field — type any portion of a revision number to filter the list to matching rows.
Script editor (top-right)Choose File opens a file browser to select the update script provided by the developer (typically a .sql file from the local filesystem). Once a file is selected, its text appears in the dark editor area for review. EXECUTE runs the script against the selected databases.
Results panel (bottom-right)Displays a pass/fail line per database after execution completes.

Process

  1. Filter the database list as needed using the Database Selection dropdown and/or the Revision Number field. Filtering narrows the list so it's easier to find the right targets without accidentally checking the wrong row.
  2. Select the databases to update by checking the box next to each row.
  3. Choose the update script using the Choose File button. The script's contents will appear in the dark editor area — review it before proceeding.
  4. Click EXECUTE to run the script against all selected databases. Wait for the run to complete; pass/fail results will populate the Results panel.
Recommendation

Limit each execution to 5 databases at a time. Running a script against too many databases in a single execution increases the chance of partial failures, makes the Results panel harder to read, and complicates rollback if something goes wrong. For larger rollouts, batch the work into multiple executions.

Best Practices
  • Always review the script contents in the editor area before clicking Execute, even if you trust the source.
  • Run scripts against a Test database first when possible, then promote to Production after verification.
  • If a script fails on one or more databases, investigate the cause before re-running on the same targets — re-executing a partially-applied script can compound the problem.
